§1 Application Overview

HelpDesk Companion is a WPF-based GUI application built in Windows PowerShell 5.1. A thin entry-point (Main.ps1) bootstraps the .NET WPF environment, loads configuration via a four-tier cascade, parses XAML files, and delegates all logic to imported PowerShell modules.

Technology Stack

ComponentTechnology
RuntimeWindows PowerShell 5.1 (64-bit required)
UI FrameworkWPF via PresentationFramework.dll (.NET Framework)
UI DefinitionXAML files loaded at runtime with {Theme_*} token-based theme injection
AD IntegrationMicrosoft ActiveDirectory PowerShell module (RSAT)
Remote ManagementWinRM / Invoke-Command, CIM/WMI, PnPUtil, quser, logoff, pnputil
ITSM IntegrationFreshservice REST API v2 (HTTPS, Basic auth with API key)
ConfigurationJSON — 4-tier cascade (UNC share → local fallback → embedded defaults → user prefs overlay)
Credential StorageWindows Credential Manager (CredRead/CredWrite via P/Invoke)
LoggingDaily CSV files on UNC share via Export-Csv with duplicate suppression
ThreadingDispatcherTimer (UI timers) + Start-Job (background operations) + BeginInvoke(Background) (lazy tab loads)

§4 ActiveDirectory.psm1

Wraps all RSAT ActiveDirectory module calls. Gracefully degrades if the AD module is unavailable. All functions return normalized PSCustomObject arrays matching the main list schema.

FunctionDescription
Get-LockedADUsersSearch-ADAccount -LockedOut enriched with Get-ADUser. Filters by FilteredUsers config list. Returns Name, DisplayValue, Type, Description, LockedOut, LastLogonDate, IsOnline, EmailAddress, LockoutTime, SamAccountName, DisplayName.
Unlock-ADUsersIterates username array, calls Unlock-ADAccount per account against the configured domain controller. Returns array of error messages for failed accounts.
Search-ADUsersWildcard filter on SamAccountName and DisplayName. Input sanitized via Sanitize-ADFilterTerm.
Search-ADComputersWildcard search + 100ms ping per result for online/offline status. Input sanitized via Sanitize-ADFilterTerm. Online/offline shown as [Online]/[Offline] prefix in DisplayValue.
Sanitize-ADFilterTermStrips LDAP special characters (* ( ) \ NUL) from user-supplied search terms before filter construction. Security control for LDAP filter injection attacks.
New-ComplexPasswordCryptographically random 16-char password with guaranteed char-class coverage (upper, lower, digit, special). Uses .NET random sourcing — not Get-Random.
Reset-ADUserPasswordSet-ADAccountPassword + Set-ADUser -ChangePasswordAtLogon. Returns $true on success. Logs result via Add-AppLog.
Get-UserDetailsGet-ADUser or Get-ADComputer with -Properties * for full attribute retrieval in the Account Details side panel.
Security — AD Filter Injection
Both Search-ADUsers and Search-ADComputers sanitize input through Sanitize-ADFilterTerm, which strips LDAP special characters before filter construction. This was the only open injection risk; it is now resolved.

§5 RemoteManagement.psm1

All remote operations use WinRM (Invoke-Command) or CIM/WMI. Time-sensitive operations wrap Start-Job with a manual polling loop to prevent UI thread blocking. All WMI hardware queries use an on-remote CleanStr sanitiser that strips XML-illegal characters before serialization — this resolves the Lenovo firmware "XML is invalid" WinRM error.

Notable UI Patterns

# (avoids circular module dependencies)
if ($State.Actions.RefreshData) { & $State.Actions.RefreshData }

# Non-blocking background query pattern (lockout source, uptime, etc.)
$job = Start-Job -ScriptBlock { ... } -ArgumentList $user
$timer = New-Object System.Windows.Threading.DispatcherTimer
$timer.Interval = [TimeSpan]::FromMilliseconds(500)
$timer.Add_Tick({
    if ($job.State -ne 'Running') {
        $timer.Stop(); $loadWin.Close()

    }
})

# Lazy tab loading — fires on first tab selection only
$tabControl.Add_SelectionChanged({
    $action = [System.Action]{
        # Fetch data on background thread
        $data = Get-RemoteData -ComputerName $comp
        # Marshal result back to UI thread
        $listView.Dispatcher.Invoke({ $listView.ItemsSource = $data })
    }
    [System.Windows.Threading.Dispatcher]::CurrentDispatcher.BeginInvoke(
        [System.Windows.Threading.DispatcherPriority]::Background, $action) | Out-Null
})

Remote Quick Actions — Runtime Injection

The Remote Quick Actions sub-menu is not defined in XAML. It is built entirely in PowerShell code within Register-ComputerUIEvents and added to the ListView ContextMenu at runtime. This allows menu items (GPResult, Uptime, PowerShell Session) to be moved from the flat context menu into the sub-menu without modifying XAML. The implementation checks for an existing sub-menu before creating a duplicate, making it safe to call Register-ComputerUIEvents multiple times.

Active Sessions Dynamic Sub-Menu

The Active Sessions menu item uses Add_SubmenuOpened rather than Add_Click. When the sub-menu opens, a Start-Job runs quser.exe against the target computer. A DispatcherTimer polls the job every 400ms. Results are rendered as nested MenuItems (user → logoff button) built entirely in code. The menu caches its last queried computer in the Tag property so repeated opens don't re-query unnecessarily.

PowerShell Bracketed Path Bug
The central config path contains square brackets ([IT Toolkit]). PowerShell treats these as wildcard metacharacters when using -Path or -Filter. All file operations on this path must use -LiteralPath or Test-Path -LiteralPath. This applies to both Get-AppConfig and GenDash.ps1. GenDash.ps1 safely uses the fast -Filter parameter by escaping the path with [System.Management.Automation.WildcardPattern]::Escape($Path) and using the -Path parameter instead of falling back to pipeline filtering.

§8 Theming System

Two complementary mechanisms depending on context:

Main Window — WPF DynamicResource

All color keys (e.g., WindowBackground, AccentFill) are set as SolidColorBrush entries in Window.Resources at runtime by the ApplyTheme scriptblock. Switching the theme at runtime updates resources in-place with no window reload. ListItemHeight and AppFontSize are also set as typed Double resources, enabling density and font size changes to take effect immediately.

Sub-Windows — Token Replacement

Dialog XAML files contain {Theme_Bg}, {Theme_Fg}, etc. Before XamlReader.Load, every token is replaced with the matching hex from Get-FluentThemeColors:

PowerShell
foreach ($key in $ThemeColors.Keys) {
    $xamlText = $xamlText -replace "\{Theme_$key\}", $ThemeColors[$key]
}

§9 PrinterManager.ps1 Architecture

Deliberately isolated as a standalone process launched via Start-Process. This prevents driver install operations from blocking or crashing the main UI thread.

1
UI_ComputerActions.psm1 calls Start-Process powershell.exe with -File PrinterManager.ps1 -ComputerName <target> -Theme <current theme> in a hidden window.
2
Script detects 32-bit process on 64-bit OS; relaunches via sysnative\powershell.exe (required for Get-PrinterDriver which fails silently in 32-bit WoW64 processes).
3
Imports CoreLogic.psm1 and RemoteManagement.psm1, reads config, calls Get-FluentThemeColors using the runtime theme, loads XAML via embedded here-string with token substitution, calls ShowDialog().
4
On load, parallel jobs check ping + port 9100 TCP health for each printer IP, with a 4-second deadline. Results are rendered as Online / Ping OK / Offline health badges in the HealthBadge column.

Driver Deployment Flow

1
Source driver folder copied to \\<PC>\C$\Temp\HDC_Drivers\ via admin share.
2
Invoke-Command runs pnputil.exe /add-driver on remote to stage into driver store.
3
Add-PrinterDriver called remotely to register with the print spooler.
4
Driver ComboBox refreshed with updated remote driver list. If the deployed driver name is found, it is pre-selected automatically.

§10 GenDash.ps1 — Dashboard Generator

GenDash.ps1 is a standalone polling script that reads CSV audit logs, queries Active Directory for currently locked accounts, and generates a self-refreshing HTML dashboard. It is not part of the main application process — it runs independently (e.g., as a scheduled task or on an IT workstation left open) and outputs a single HTML file to a network path accessible via browser.

Architecture

  • Polls the log directory every 5 seconds by computing a file signature (name + LastWriteTime + Length). Only regenerates the HTML when the signature changes.
  • Reads all CSV log files using System.IO.File.Open with FileShare.ReadWrite to avoid locking conflicts with the running application. Retries on transient lock failures.
  • Concatenates all log files into a single CSV string, encodes as Base64 UTF-8, and injects it into the HTML template as a JavaScript variable. This prevents all special-character escaping issues (backticks, quotes, non-ASCII) from breaking the browser's JavaScript engine.
  • Queries AD for currently locked accounts via Search-ADAccount -LockedOut and serializes the result as compact JSON for injection into the dashboard's AD lockout table.
  • Outputs to a configured UNC path (\\vm-simplify\custom\fileuploads\acctdashboard.html). The HTML auto-refreshes in the browser every 120 seconds.

Adding a New Module

1
Create Modules\UI_NewFeature.psm1 with a Register-NewFeatureUIEvents($Window, $Config, $State) function.
2
Add Import-Module "$PSScriptRoot\UI_NewFeature.psm1" -Global -Force -DisableNameChecking to UIEvents.psm1.
3
Call Register-NewFeatureUIEvents inside Register-MainUIEvents in UIEvents.psm1.
4
Export all public functions with Export-ModuleMember -Function *.

Adding a New Dialog

1
Create XAML in UI\Dialogs\ using {Theme_*} tokens for all color values.
2
$win = Load-XamlWindow -XamlPath (...) -ThemeColors (Get-FluentThemeColors $State)
3
Wire controls: $win.FindName('name').Add_Click({...}.GetNewClosure())
4
Set $win.Owner = $Window and call Show-CenteredOnOwner -ChildWindow $win -OwnerWindow $Window before ShowDialog() to correctly center on the active monitor in multi-monitor setups.

Adding a New Context Menu Item

1
Declare the MenuItem in the XAML context menu with x:Name and Visibility="Collapsed" as the initial state.
2
Locate it in the module: $ctx = $Window.FindName('ctxYourItem')
3
Add a visibility setter in the ContextMenuOpening handler in the relevant UI module to show/hide based on selected item type.
4
Wire the click handler: $ctx.Add_Click({ }.GetNewClosure())
5
For dynamically injected items (not in XAML), create a new System.Windows.Controls.MenuItem in code and add it to $lvData.ContextMenu.Items. Check for existing items before adding to avoid duplicates on re-registration.

Adding a New Theme Token

  • Add the key to both LightModeColors and DarkModeColors in HDCompanionCfg.json.
  • Map it in Get-FluentThemeColors return hashtable in CoreLogic.psm1.
  • Add DynamicResource binding in MainWindow.xaml Resources section for the main window.
  • Use {Theme_YourKey} in any XAML loaded via Load-XamlWindow.
  • Update the token documentation in Tech_Docs.html §8.
